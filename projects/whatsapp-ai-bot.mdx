## Problem

Wiring a business up to WhatsApp usually means writing the same webhook boilerplate every time — signature validation, message deduplication, session handling — and coupling it tightly to whichever AI backend that particular business happens to use. I wanted a webhook layer that stayed the same regardless of what was answering behind it, so integrating a new business meant swapping one URL, not rewriting the integration.

## Architecture / Approach

- **A Meta Cloud API webhook** handles the `GET` verification handshake and validates every inbound `POST` against `X-Hub-Signature-256` using the app secret, so spoofed requests are rejected before they reach any business logic.
- **A fixed request/response contract** — `{session_id, message}` in, `{message}` out — is the only thing a chat backend needs to implement. Swapping backends means changing `CHAT_BACKEND_URL`; nothing else in the webhook layer changes.
- **File-backed idempotency** deduplicates inbound message IDs, since Meta can redeliver the same webhook event more than once.
- **A local Gemini shim** implements the same contract for development, so the webhook layer can be built and tested without needing a production AI backend running yet.
- **A policy/guardrail layer** (JSON-configured) enforces an opt-in welcome menu before the AI responds, keeps replies inside WhatsApp's 24-hour customer-service window, and defines an escalation path to a human agent — the operational rules that matter more than the AI response quality for a real deployment.

## My Contribution

I designed and built the entire system: the webhook signature/idempotency layer, the generic chat-backend contract that decouples the integration from any specific AI provider, the Gemini shim used for local development, and the policy layer (interactive first-contact menu, stored-first-question handling, feedback prompts, typing indicators). I also wrote the full operational documentation — architecture notes, a Meta dashboard setup walkthrough, a variable-to-source mapping for every credential, and a troubleshooting guide — specifically so the template could be handed to a new business integration without me in the loop.

## Outcome

The result is a webhook integration that's genuinely reusable — pointing `CHAT_BACKEND_URL` at a new service and updating a few `BUSINESS_*` environment variables is enough to stand up a new WhatsApp-based assistant without touching the webhook or security code. Keeping the contract this narrow (one JSON shape in, one JSON shape out) turned out to be the single decision that made every later integration cheaper.
