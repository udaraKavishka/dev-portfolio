## Problem

Automated paraphrasing tools rewrite text for you and get the tone wrong more often than not. What I actually wanted while editing my own writing was control: see one sentence at a time with surrounding context, rewrite it myself, and move on — without losing my place or my work if the browser crashed. That's a deceptively hard product to build well: correct sentence splitting alone breaks on abbreviations, decimals, citations, and quotes long before you get to auto-save, exports, or performance on a 30,000-sentence document.

## Architecture / Approach

- **A configurable sentence-splitting engine** runs in three modes (strict/balanced/loose) with four domain profiles (academic, legal, technical, medical), each tuned to a different set of abbreviation and citation patterns (`Dr.`, `et al.`, `v.`, `Inc.`, decimals, ellipses) so the same core engine doesn't mis-split a citation the way it would a casual sentence.
- **A Web Worker** runs the splitting engine off the main thread, so a 30,000-sentence document doesn't freeze the UI — the app stays responsive while it processes in the background.
- **Auto-save with crash recovery** snapshots the editing session to `localStorage` every 30 seconds with a 7-day retention window, so a browser crash mid-edit doesn't cost hours of work.
- **A Python (FastAPI) extraction backend** handles PDF/DOCX parsing server-side as a fallback path, with an OCR retry mode (`pytesseract`) for scanned PDFs that the client-side `pdf-parse` library can't read — kept as a separate service behind a CORS-configured API rather than bundled into the frontend, so it can be deployed and scaled independently.
- **Multi-format export** (TXT, DOCX, Markdown) preserves the original paragraph structure rather than flattening everything to plain text.

## My Contribution

I designed and built the whole system solo: the splitting engine and its 15+ edge-case rules, the Web Worker integration, the auto-save/crash-recovery layer, readability metrics (Flesch Reading Ease/Grade Level), and the multi-format export pipeline on the frontend; and the FastAPI extraction service with OCR fallback, CORS configuration, and Supabase-backed logging on the backend. I also set up a CI workflow that lints and tests both the frontend splitter suite and the backend service on every push, and wrote a deterministic test suite (18 cases, 100% pass rate) covering the abbreviation and citation edge cases the splitter has to get right.

## Outcome

ParaWrite runs in production at parawrite.udaradev.me, processing sentences at roughly 88,000/second with no UI freeze even on large documents. Building the domain-profile splitting logic taught me more about the actual shape of "what counts as a sentence" than I expected going in — abbreviations, citations, and decimals turn out to be the majority of the hard cases, not the exception.
