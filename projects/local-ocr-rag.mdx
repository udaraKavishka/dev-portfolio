## Problem

Most RAG demos assume a hosted embedding API, a hosted LLM, and English-only documents. None of that holds for a lot of real use cases in Sri Lanka: documents mix Sinhala, Tamil, and English, sending them to a third-party API isn't always acceptable, and "it works with an OpenAI key" isn't the same as "it works with zero cloud dependencies." I wanted a RAG system that could sit entirely on one machine — no API keys, no data leaving the device — and still handle scanned documents and three languages correctly.

## Architecture / Approach

DocMind is built as six independent, observable layers:

- **L1 — File watcher**: a Go binary (`fsnotify`) and a Python thread (`watchdog`) both monitor `watched_docs/`, so dropping a file in triggers ingestion automatically within seconds.
- **L2 — OCR microservice**: a standalone FastAPI service (port 8765) running Tesseract 5 with Sinhala, Tamil, and English language packs, with OpenCV/Pillow preprocessing (grayscale → Otsu threshold → denoise) ahead of OCR to improve accuracy on scanned pages.
- **L3 — Parsing and chunking**: format-specific parsers (PDF via `pdfplumber` with OCR fallback, PPTX, DOCX, images, CSV/Excel) feed a parent-child chunker — 1024-token parent chunks for context, 256-token child chunks for precise retrieval, with 50-token overlap — using `tiktoken` for consistent counting across languages.
- **L4 — Hybrid retrieval**: dense vector search (ChromaDB, `nomic-embed-text` embeddings) runs alongside sparse BM25 keyword search, fused with Reciprocal Rank Fusion rather than relying on embeddings alone — this matters specifically for multilingual text, where dense embeddings alone can miss exact-term matches that BM25 catches.
- **L5 — Reranking and agent loop**: the fused top-20 results are reranked by a multilingual cross-encoder (BGE-reranker-v2-m3), then a LangGraph agent runs planner → retriever → generator → critic nodes — the critic checks whether the generated answer is actually grounded in retrieved context, and re-runs retrieval (up to 3 times) if it isn't, rather than returning an ungrounded answer.
- **L6 — Frontend**: a Streamlit chat interface shows the answer with source citations, a language filter, a model switch (qwen3:8b for quality, gemma3:4b for speed), and a debug mode exposing chunk-level RRF/rerank scores.

Every model in the pipeline runs locally through Ollama, sized to fit a 12GB RAM budget end to end — the README documents the exact memory footprint of each component (generation model, embeddings, reranker, indexes) because staying under that budget was a real design constraint, not an afterthought.

## My Contribution

I designed and built the full six-layer pipeline solo, in both Python and Go: the dual-language file watcher (Go binary for standalone use, Python watchdog thread inside the Streamlit app), the OCR microservice with multilingual preprocessing tuned for Sinhala/Tamil accuracy, the parent-child chunking strategy, the hybrid retrieval layer (dense + BM25 + RRF fusion), the reranking step, and the LangGraph agent with its self-critique retry loop. I also wrote the per-layer documentation (`docs/L1_*.md` through `L6_*.md`) as I built each layer, and a `tmuxp` session config so the five required services (OCR, watcher, Streamlit, Ollama, shell) start with one command instead of five manual terminal setups.

## Outcome

DocMind is an actively developed, fully local RAG system — over 400 commits in — that answers questions across three languages with zero API keys and zero cloud calls, staying within a 12GB RAM budget on commodity hardware. Building the multilingual hybrid retrieval layer was the part that most changed how I think about RAG: dense embeddings alone are not enough once documents mix scripts and languages, and BM25 plus a real reranker isn't optional at that point, it's load-bearing.
