## Problem

This site was originally built with a headless CMS (Sanity) driving the blog. That's a reasonable choice for a team, but for a single-author portfolio it added a hosted dependency, an extra dashboard to maintain, and a network round-trip for content that changes rarely and could just live in the repo. I wanted writing a blog post or a project case study to mean "add a file and push" — no CMS login, no schema to keep in sync, and full content history in git.

## Architecture / Approach

- **Local MDX files** (`posts/*.mdx`, `projects/*.mdx`) are the entire content layer — frontmatter via `gray-matter`, rendered server-side with `next-mdx-remote/rsc` and `remark-gfm`.
- **`lib/posts.ts` and `lib/projects.ts`** parse, sort, and expose the content with a small typed interface (slug, title, date, excerpt, tech stack), read straight off the filesystem at build time — no database, no API calls.
- **Shared MDX components** (`lib/mdx-components.tsx`) give every post and case study consistent code blocks (syntax-highlighted) and images, so writing content doesn't mean re-solving formatting per page.
- **SEO is generated, not hand-maintained** — `sitemap.ts` walks the same `getAllPosts()`/`getAllCaseStudies()` functions to build the sitemap, and `StructuredData.tsx` emits JSON-LD (Article/CaseStudy/Breadcrumb schemas) per page so search engines get structured metadata without a separate content pass.
- **Project case studies** (`/projects/[slug]`) reuse the exact same MDX pipeline as the blog — same frontmatter shape, same rendering path — rather than building a second content system for a second content type.

## My Contribution

I designed and built the entire site solo, including migrating the blog off Sanity to local MDX (removing the CMS dependency, its API client, and the network fetch it required on every page load), building the case-study pipeline as an extension of the existing blog pattern rather than a new system, and wiring up the SEO layer (sitemap, structured data, Open Graph metadata) to derive from content automatically instead of being maintained by hand per page. I also built blog search, related-posts, and pinned-post functionality on top of the same content layer.

## Outcome

The site now has zero external content dependencies — every post and case study is a file in the repo, versioned in git, deployable with a normal `git push`. The CMS migration cut a full external service and its API surface out of the site's critical path, and the case-study feature (the one generating this very page) took a fraction of the effort it would have if it hadn't been able to reuse the blog's existing MDX parsing, rendering, and SEO plumbing.
