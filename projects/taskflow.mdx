## Problem

TaskFlow was a university group project: a Trello-style task manager with boards, lists, cards, and full collaboration features (labels, checklists, due dates, attachments, comments). With a two-person team, the work needed a clean split — rather than both of us touching the same layers, I took ownership of the entire backend API so the frontend developer could build against a stable, versioned contract instead of the two of us stepping on each other's routes.

## Architecture / Approach

- **Express.js + MongoDB/Mongoose** provide the REST API and schema for boards, lists, cards, and every sub-resource attached to a card (labels, members, checklists, comments, attachments, due dates).
- **JWT authentication** is applied through a `verifyToken` middleware registered globally across protected routes, so every controller can trust `req.user` without re-checking auth per endpoint.
- **A controller/service split per resource** — each of the nine resource types (board, list, card, label, member, checklist, comment, attachment, date) gets its own controller and service pair, keeping request handling separate from the Mongoose queries and business rules underneath it.
- **React + Redux Toolkit** on the frontend consumes this API for drag-and-drop boards (`react-beautiful-dnd`), styled with Material UI and styled-components.

## My Contribution

I built the backend from the ground up: user registration/login with JWT and bcrypt password hashing, the full board → list → card hierarchy with ordering logic (reordering lists and cards without breaking sibling positions), and every card sub-feature — labels, member assignment, checklists with completable items, due dates, comments, and attachments — each as its own routed resource with dedicated controllers and services. I also handled the recurring class of bugs that comes with a card-based UI: fixing broken axios calls from the frontend, correcting list/card reordering after a drag-and-drop, and getting global token verification working consistently across every route.

## Outcome

A complete, working task-management REST API with 9+ resource types, JWT-secured throughout, that the frontend could build a full drag-and-drop board experience against. Splitting ownership this way — one person per layer instead of both people touching everything — meant the API contract stayed stable while the UI was being built, which is the same reason I reach for a clear frontend/backend split on any team project since.
