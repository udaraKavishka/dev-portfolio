## Problem

Dijkstra's algorithm alone finds the shortest path by distance, but distance isn't the same as travel time — footpaths get crowded at different times, some routes are longer but faster, and a static shortest-distance path can be the wrong recommendation in practice. This was a five-person university team project for the University of Vavuniya campus, and my part was making the pathfinding travel-time-aware rather than distance-only.

## Architecture / Approach

- **A graph model of the campus** (`NetworkX`) represents buildings and paths as nodes and weighted edges.
- **A Random Forest regression model** (`scikit-learn`) predicts travel time per edge from historical/simulated movement data, rather than using raw physical distance as the edge weight.
- **Dijkstra's algorithm** runs shortest-path search over the graph using the model's predicted travel times as edge weights, so the "shortest" path optimizes for time, not just distance.
- **A Tkinter + Pillow desktop GUI** lets a user enter source and destination, then draws the computed path as a red line over the campus map image with the total distance/time shown.

## My Contribution

I was in charge of the machine learning side of the project: training the Random Forest model that predicts travel time for each edge in the campus graph, and integrating its output as the edge-weighting function that Dijkstra's algorithm actually searches over — the piece that turns "shortest distance" into "fastest predicted route." The rest of the team built the graph data, the GUI, and the map integration around that.

## Outcome

The system was nominated for Best Final Year Project at the University of Vavuniya. It's a compact example of a pattern I still reach for: don't optimize a search algorithm over a naive metric (raw distance) when you can train a small model to predict the metric you actually care about (travel time) and feed that into the same well-understood algorithm instead.
