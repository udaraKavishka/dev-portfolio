The error looked like this:

```
alembic.util.exc.CommandError: Can't locate revision identified by 'f7a2c9d1b3e4'
```

If you've used Alembic (or Flask-Migrate, which wraps it), you may know the cold feeling this message produces. The database says it's at revision f7a2c9d1b3e4. Alembic scans the migration files and finds no such revision. The two sources of truth have diverged, and neither one can be casually rewritten.

In our case the cause was ordinary: a migration file had been deleted during branch cleanup, after it had already been applied to a database. The file was the only record of what that revision was. Git had it, thankfully, and restoring the file fixed everything in one commit. But sitting with a database that referenced a phantom revision made the mental model click in a way documentation never had.

Alembic tracks state in two places. The `alembic_version` table in your database holds the ID of the currently applied revision. The files in `versions/` form a chain, each one naming its parent. Every Alembic operation works by walking that chain between where the database is and where the files say it could go. Delete a file from the middle (or from the tip, if a database points at it) and the walk fails. Migration files aren't scaffolding you can prune. They're the ledger.

The routine I now run before generating any migration:

```bash
alembic current   # what the database thinks
alembic heads     # what the files think
```

Thirty seconds. If `current` matches the single head, generate away. Two heads mean parallel branches created migrations independently, which wants an explicit merge revision, not an autogenerate on top of the mess. A `current` that names a missing revision means someone deleted history, and you want to know that now, not after stacking new work on a broken chain.

Related habits that have each earned their place: always read an autogenerated migration before applying it, because autogenerate reflects every accidental model change, not just the one you intended. Test the downgrade path at least once (upgrade, downgrade, upgrade) while the migration is fresh, since a broken downgrade discovered during an incident is the worst time to discover it. And treat migration files in code review with the same seriousness as application code, because unlike application code, a bad migration executes against production data exactly once, with no retry.

The chain only stays trustworthy if every link stays in the repo. Guard the files.
