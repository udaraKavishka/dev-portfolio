**Why FastAPI and RAG?**

****

**FastAPI** is a modern Python web framework that makes it easy to build APIs with automatic documentation, type validation, and high performance. It's used by companies like Uber, Netflix, and Microsoft for building production APIs.

**RAG (Retrieval-Augmented Generation)** combines document search with AI generation. Instead of the AI guessing from its training data, RAG first searches your documents for relevant information, then uses that context to generate accurate answers.

#### Create embedding script

You've created your knowledge document with Kubernetes content. To make this content searchable by your RAG system, we need to convert it into embeddings.

> **What are embeddings?**
Embeddings are numerical representations of text that capture meaning. Words with similar meanings are placed close together, while unrelated words are far apart. For example, "container" and "Docker" would have embeddings that are mathematically similar. This allows the computer to understand relationships between words based on their meaning, not just spelling.

**How do embeddings work in RAG?**
When you ask "What is Kubernetes?", the system:

- Converts your question into an embedding (a list of vectors)
- Compares it to embeddings of all text in your knowledge base
- Finds the text with the most similar embedding (meaning)
- Uses that text as context for generating the answer

This is much more powerful than keyword matching - it understands meaning!

**Create a new file embed .py**

```python
import chromadb

client = chromadb.PersistentClient(path="./db")
collection = client.get_or_create_collection("docs")

with open("k8s.txt", "r") as f:
text = f.read()

collection.add(documents=[text], ids=["k8s"])

print("Embedding stored in Chroma")
```

Let's break down each key line:

> **import chromadb **- Imports the Chroma library so we can use it in our script.

> **client = chromadb.PersistentClient(path="./db") **- Creates a Chroma client that stores data permanently in a new ./db folder on your computer. The PersistentClient means the data will be saved to disk, so it persists even after you close the program.

> **collection = client.get_or_create_collection("docs")** - Gets an existing collection named "docs" or creates a new one if it doesn't exist. A collection is like a folder that holds your documents and their embeddings. You can have multiple collections for different types of documents.

> **with open("k8s.txt", "r") as f:** - Opens the k8s.txt file in read mode. The with statement automatically closes the file when done, which is a best practice in Python.

> **text = f.read()** - Reads all the content from the file and stores it in the text variable.

> **collection.add(documents=text, ids="k8s")** - Adds the text to the collection with the ID "k8s". Chroma automatically converts the text into embeddings (numerical representations) when you add it. The ids parameter gives each document a unique identifier so you can reference it later.

>  **print("Embedding stored in Chroma")** - Prints a confirmation message to let you know the process completed successfully.

Let's create the main API application that will handle incoming questions, search your knowledge base in Chroma, and generate answers using tinyllama. We'll use FastAPI to create the API app.

> **What is FastAPI?**
FastAPI is a modern Python web framework for building APIs. It's designed to be fast, easy to use, and automatically generates interactive documentation. Companies like Uber, Netflix, and Microsoft use FastAPI for their APIs.

> **Why use FastAPI?**
Normally, when building an API from scratch, you'd need to deal with a lot of technical details: your code needs to listen for specific requests from other apps, understand the data format, make sure the right information is received, and then send back results in a way other programs understand.

FastAPI takes care of a lot of this background work for you: it handles how requests come in and go out, checks if incoming information is correct, and returns answers in the right format. This means engineers (like you!) can focus on describing what should happen rather than all the complicated communication and data handling rules that usually go into building an API.

#### How does this RAG API work?


This FastAPI app creates a web server with one endpoint: /query. Here's how your API works, step by step:

1. Question arrives: Someone sends a question to your API at /query
1. Search your documents: Chroma searches through your knowledge base to find text that matches the question's meaning
1. Get matching text: Chroma returns the most relevant information from your documents (this is called "context")
1. 4Generate answer: The question and the matching text are sent together to tinyllama, which creates an answer
1. Send back the answer: The answer is sent back to whoever asked the question

This is the complete RAG pattern: Retrieve relevant information, then Augment the LLM's generation with that information. FastAPI handles the HTTP layer, making this accessible as a web service.

### Add dynamic content to your knowledge base

> **What is dynamic content?**
Dynamic content means information that can be added, updated, or removed while your application is running, without having to restart it or manually edit files. Instead of only being able to add content by editing k8s.txt and re-running embed.py, you'll be able to add new information through an API call.

Why is this useful?
In production systems, knowledge bases need to be updated frequently as new information becomes available. Having an API endpoint to add content means:

- Other applications can update your knowledge base automatically
- Users can add information through a web interface
- Your system can stay up-to-date without manual intervention
- You can expand your knowledge base incrementally

**What does this code do?**
This endpoint:

> **Why generate a unique ID?**
Each document in Chroma needs a unique identifier. We use uuid (Universally Unique Identifier) to generate a random, unique ID for each piece of content we add. This ensures we can reference specific documents later if needed.

#### Can the API still hallucinate details I didn't give it?


Yes, the API can still add details that weren't in your knowledge base! Here's why:

Small models like tinyllama are trained to generate coherent, detailed responses. When given minimal context, they often "fill in the blanks" to create more complete-sounding answers, even if those details aren't in your knowledge base.

#### How can you reduce hallucination?

- Provide more detailed context in your knowledge base (the more specific information you give, the less the model needs to invent)
- Use larger, more capable models (like Llama 3.3 70B) that are better at sticking to provided context
- Improve your prompts to explicitly tell the model to only use information from the context
- Add validation in your API to check if responses match your knowledge base

RAG significantly reduces hallucination by providing relevant context, but it doesn't eliminate it completely - especially with smaller models. This is an active area of research in AI!
